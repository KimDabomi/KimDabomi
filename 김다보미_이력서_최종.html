PHP 5.4 기반 주문·결제 시스템은 주문, 배송, 적립금 로직이 레거시 데이터베이스 직접 조회 방식으로 결합되어 있어 변경 시 장애 위험이 높았습니다. 동시에 물류 출고 메일과 적립금 만료 처리는 수작업 의존도가 높아 운영 공수가 지속적으로 발생했습니다. 기존 관리자·사용자 화면을 유지한 채 신규 API 구조로 이관해야 했기 때문에, 기능 교체보다 신·구 시스템의 공존과 데이터 정합성 유지가 핵심 과제였습니다.

  • 출고 데이터 조회, 출고 명세서·엑셀 생성, 환경별 수신자 관리, 메일 발송을 하나의 관리자 API 흐름으로 구성하고, 발송 전 요약 조회 API를 별도로 제공해 운영자가 수취인·상품·수량·가격을 검증할 수 있도록 했습니다.

Spring Boot · PostgreSQL · MySQL · Next.js

배경 및 해결 과제

일반 로그인, 소셜 로그인, 레거시 관리자 인증이 서로 다른 구조로 운영되어 회원 생성과 계정 연동 로직이 파편화되어 있었습니다. 신규 인증 체계를 도입하면서도 비밀번호가 없는 소셜 회원, 기존 레거시 식별자, 휴면 계정과 같은 예외 케이스를 흡수해야 했고, 카카오와 애플의 서로 다른 OAuth 응답 방식까지 웹 클라이언트에서 일관되게 처리해야 했습니다.

기술적 의사결정

  • 외부 OAuth 인증 결과를 내부 회원 도메인과 매핑한 뒤 자체 보안 토큰을 발급하도록 구성하고, 기존 회원 로그인과 신규 회원 등록을 단일 소셜 인증 엔드포인트에서 처리했습니다.
  • Apple OAuth의 response_mode=form_post 특성을 고려해 Next.js API Route에서 authorization code를 수신한 뒤 로그인 페이지로 전달하는 중계 구조를 설계했습니다.
  • 비밀번호가 없는 소셜 회원도 레거시 회원으로 등록할 수 있도록 신규 회원 정보 업데이트와 레거시 저장 책임을 분리하고, 레거시 식별자 생성은 데이터베이스 시퀀스로 통일했습니다.
  • OAuth 장애 원인 추적을 위해 프론트 요청과 백엔드 로그에 redirect_uri를 포함하고, provider 오류 유형별로 예외와 로그 레벨을 구분했습니다.

자발적 개선

카카오 인증 코드 중복 오류를 단순 재현하는 데 그치지 않고, 요청이 정상 클라이언트에서 발생했는지 임의 요청인지 구분할 수 있도록 redirect_uri를 관측 지표로 추가하고 외부 인증 요청·응답 로깅 구조를 개선했습니다.

결과

파편화된 인증 흐름을 일원화하고 카카오·애플·일반 회원가입을 하나의 회원 도메인으로 통합했습니다. 전년 동기 대비 신규 회원가입 유입률이 163.2% 증가했으며, 소셜 회원과 레거시 회원 간 데이터 불일치 가능성을 줄였습니다.

4. 대량 기업회원 가입의 병목을 제거하고 식별자 생성 책임을 DB로 이관

Spring Boot · JPA/Hibernate · PostgreSQL
성능/정합성

배경 및 해결 과제

B2B 사업 확장으로 수십~수백 명의 기업회원을 한 번에 등록해야 했지만, 초기 구현에서는 50명 등록에 17초 이상이 소요되었습니다. 대량 가입 로직은 회원, 레거시 회원, 약관, 마일리지 데이터를 함께 생성하므로 일부 단계의 실패가 전체 데이터 정합성에 영향을 줄 수 있었고, 단순 병렬화만으로는 해결하기 어려운 구조였습니다.

기술적 의사결정

  • 프로파일링을 통해 회원 생성 루프마다 레거시 식별자 시퀀스를 DB에서 조회하는 구조가 네트워크 I/O와 커넥션 사용을 증가시키는 핵심 병목임을 규명했습니다.
  • old_member_idx 컬럼에 sequence default를 지정해 식별자 생성 책임을 애플리케이션에서 DB 엔진으로 이관하고, DB sequence의 원자성을 활용해 동시성 안정성을 확보했습니다.
  • JPA/Hibernate가 null 컬럼까지 insert SQL에 포함해 DB default가 동작하지 않는 문제를 분석하고, @DynamicInsert, @Generated(event = INSERT), updatable = false를 적용했습니다.
  • 저장 전에 생성한 식별자를 여러 중간 객체에 복사하지 않고, 회원 저장 이후 DB에서 확정된 레거시 회원 데이터 인덱스를 기준으로 레거시 등록 DTO를 생성하도록 처리 순서를 재설계했습니다.
  • 대량 가입 요청·응답 객체는 생성 이후 상태 변경을 최소화하는 방식으로 구성해 청크·병렬 처리 중 중간 데이터 오염 가능성을 낮췄습니다.

자발적 개선

요구된 대량 가입 기능 구현 이후 처리 시간이 기대 수준에 미치지 못한다는 문제를 스스로 발견하고, 반복 DB 호출과 JPA insert 메커니즘까지 추적해 구조적 개선안을 제안·적용했습니다.

결과

50건 처리에 약 17초가 소요되던 흐름을 개선해 100건도 약 15초에 처리하도록 최적화했습니다. 1건당 평균 처리 시간 기준 약 56% 성능 향상을 달성하고, 가입·레거시 동기화 간 데이터 불일치 가능성을 줄였습니다.

5. 운영 공수를 줄이고 장애 대응 속도를 높이는 모니터링·배치 구조 개선

Spring Batch · AWS Lambda · Sentry · Slack · PostgreSQL · MySQL
운영안정화

배경 및 해결 과제

배송 정보와 기업회원 등록은 수작업 비중이 높았고, 장애는 개별 서버 로그를 확인해야 해 인지와 원인 분석이 늦었습니다. 또한 PostgreSQL과 레거시 MySQL 간 배치에서는 이모지 데이터 적재 오류가 발생했으며, AWS Lambda 기반 Spring Batch는 warm start 상황에서 간헐적으로 Out Of Memory(OOM)가 발생했습니다. 단순 재시작이나 메모리 증설이 아니라 운영 구조 자체를 개선해야 했습니다.

기술적 의사결정

  • 배송 정보와 기업회원 등록을 청크 단위 대량 처리 API로 전환하고, 중복·기존 데이터 검증, 최대 처리량 제한, 건별 성공·실패 집계를 적용했습니다.
  • 중복 운송장처럼 운영 중 예상 가능한 예외는 경고로 분류하고 건너뛰도록 해, 실제 장애와 업무상 예외가 동일한 Sentry 경보로 쌓이지 않도록 했습니다.
  • Sentry 기반 중앙 예외 수집과 GlobalExceptionHandler 중심 공통 로깅 구조를 구축하고, 요청 URI와 환경 정보를 포함한 로그를 Slack 알림 흐름과 연계했습니다.
  • PostgreSQL에서 정상 저장되던 이모지 데이터의 MySQL 적재 실패를 문자셋 차이로 진단하고, MySQL JDBC 연결 옵션을 환경변수화해 배포 환경별 charset 설정을 주입할 수 있도록 했습니다.
  • Lambda 호출마다 SpringApplication.run()이 실행되어 warm start 컨테이너에서 ApplicationContext가 중복 생성되는 것을 OOM 원인으로 규명하고, 싱글톤 컨텍스트 캐싱 구조로 기존 컨텍스트를 재사용했습니다.
  • 배치 실행 인수의 민감정보를 마스킹하고, 실패 로깅·예외 전파와 컨텍스트 재사용 검증 테스트를 추가했습니다.

자발적 개선

운영자가 반복 수행하던 배송·회원 등록 업무를 시스템화하고, 단순 장애 복구에 그치지 않고 Sentry 경보 노이즈, Lambda warm start, DB 문자셋 차이처럼 반복 장애를 만드는 근본 원인을 추적해 구조 개선으로 연결했습니다.

결과

반복 운영 업무를 대량 처리 API로 전환하고, Sentry–Slack 기반 장애 감지·전파 흐름을 마련했습니다. Lambda warm start 환경의 OOM 원인을 제거해 배치 가용성을 높였으며, 불필요한 장애 경보를 줄여 실제 장애 대응 집중도를 개선했습니다.
